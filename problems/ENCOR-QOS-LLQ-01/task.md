# ENCOR-QOS-LLQ-01: WAN 輻輳で途切れる音声を QoS で救え（効果実測つき）

難易度: ★★★★☆（4/6） / 配点: 100点 / 対象: ENCOR (350-401) QoS

## シナリオ

あなたは 2 拠点を WAN で結ぶ企業のネットワーク担当者。
キャリアとの契約帯域は **2Mbps**（物理リンクは高速だが、**契約超過分はキャリア網で
無条件に破棄される**。破棄のされ方は制御できない）。

最近、拠点1 から拠点2 へ大容量データ転送が走るたびに
**IP 電話（DSCP EF でマーキング済み）が途切れる**とクレームが出ている。

```
PC01 ── RT01 ──[ WAN 契約2Mbps ]── RT02 ── PC02
拠点1(192.168.10.0/24)              拠点2(192.168.20.0/24)
 .10     .1      10.1.12.0/30        .1      .20
        E0/0  E0/1          E0/1  E0/0
```

- PC01 / PC02: Ubuntu（ログイン `suzuki` / `CCNP`）。iperf3 導入済み。
  PC02 では iperf3 サーバが常駐起動している（あなたが起動する必要はない）。
- RT01 / RT02: IP アドレス・経路は設定済み（**変更禁止**）。
- 音声は端末側で DSCP EF にマーキング済みという前提（本ラボでは
  `ping -Q 184` = DSCP EF で音声パケットを模擬する）。

## 要件

1. RT01 の WAN 出力を、**全トラフィック合計で契約帯域 2Mbps に収まる**よう整形せよ
   （キャリアに破棄させるのではなく、自分の管理下で帯域を守ること）。
2. その上で、音声（DSCP EF）は**輻輳中でも損失なく・低遅延**で転送されること。
   音声に割り当てる優先帯域は **256kbps** とする。
3. 設定変更は **RT01 のみ**（RT02・PC の設定変更は禁止。既存のアドレス・経路も変更禁止）。
4. class-map / policy-map 等の**名前は任意**（採点は名前でなく状態と効果を見る）。

## 体感手順 — 必ず「実装前」と「実装後」の両方を測定して記録すること

この問題の目的は、QoS の効果を**数値の変化**として自分の目で確かめることにある。
PC01 に SSH（またはコンソール）でログインし、以下を測定せよ。

**測定A: TCP スループット**
```
iperf3 -c 192.168.20.20 -t 10
```

**測定B: 輻輳を起こし、その最中の音声品質を見る**（ターミナル2本 or `&` を使う）
```
# 1本目: 大容量データ転送を模擬（UDP 5Mbps を 40 秒間）
iperf3 -c 192.168.20.20 -u -b 5M -l 1200 -t 40

# 2本目: 転送中に「音声」(DSCP EF) の ping — RTT と loss を記録
ping -Q 184 -c 25 -i 0.2 192.168.20.20

# 2本目つづき: 比較用に「普通の」ping も記録
ping -c 25 -i 0.2 192.168.20.20
```

**記録表**（実装前後で埋める）:

| 測定 | 実装前 | 実装後 |
|------|--------|--------|
| A: TCP スループット | Mbps | Mbps |
| B: 輻輳中の EF ping (RTT avg / loss%) | ms / % | ms / % |
| B: 輻輳中の 普通 ping (RTT avg / loss%) | ms / % | ms / % |

RT01 側の観測には `show policy-map interface Ethernet0/1` が使える。
実装後の数値が「実装前と何が・なぜ変わったか」を自分の言葉で説明できたら完成。

## 採点

- 採点は自動（構成の状態 + **効果の実測**）。採点器も PC01 から iperf3/ping を流す。
- 目安時間: 40〜60 分。
