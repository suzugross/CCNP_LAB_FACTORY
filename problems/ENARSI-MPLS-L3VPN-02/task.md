# 問題 ENARSI-MPLS-L3VPN-02 : MPLS L3VPN 応用 — PE-CE OSPF 化・広告制御・MSS 調整（難易度4）

## シナリオ
ENARSI-MPLS-L3VPN-01 で構築した L3VPN サービス（CUST_A / CUST_B）は**稼働中**です。
コア（OSPF・LDP・MP-BGP VPNv4・VRF 収容）は構成済み・正常動作しています。

顧客 2 社から**契約変更チケット**が届きました。SP 側エンジニアとして PE を変更してください。

> **チケット 1（両顧客共通）— PE-CE ルーティングの OSPF 化**
> 今後サイト内セグメントが増えるため、スタティックをやめ **OSPF による動的経路交換**に
> 切り替えたい。CE 側は OSPF（プロセス 1, area 0）の設定・稼働を開始済み。
>
> **チケット 2（両顧客共通）— 広告は必要最低限に**
> CE には**機器管理用プレフィックス（10.99.0.0/16 内の /32）**が設定されており、OSPF にも
> 乗っている。これは**各サイトと収容 PE までに閉じるべき**もので、**VPN をまたいで対向
> サイトへ広告してはならない**。VPN で運ぶのは**サイト LAN（172.16.0.0/16 内）のみ**とすること。
>
> **チケット 3（両顧客共通）— サイト間の大容量転送がハングする**
> サイト間で大きなファイル転送が止まる事象が報告されている（小さい通信は正常）。
> MPLS 網の経路 MTU が原因と判明した。**PE の CE 向けインタフェースで TCP MSS を 1452**
> に調整して対処すること。

```
 CUST_A CE1                                          CUST_A CE2
   RT04 ---\                                        /--- RT05
            RT01 ========= RT02 ========= RT03
   RT06 ---/ (PE1)          (P)          (PE2) \--- RT07
 CUST_B CE1                                          CUST_B CE2
            └──────── SPコア (構成済み) ───────┘
```

## 構成（初期状態で投入済み）

### コア（構成済み・正常動作）
- OSPF 1 / LDP（コアのみ）、PE1–PE2 の iBGP VPNv4（AS 65000, Lo0 間）、
  VRF CUST_A（RD 65000:100 / RT 65000:100）・CUST_B（RD 65000:200 / RT 65000:200）
  定義済み、CE 向け IF は VRF 収容・IP 設定済み。アドレスは 01 と同一。
- **PE-CE のルーティング（旧スタティック）は撤去済み** = 現在サイト間は不通。
  BGP の `address-family ipv4 vrf` も未設定（あなたが構成する）。

### CE（RT04〜RT07 = 顧客管理機器・設定済み・変更禁止）
各 CE は OSPF プロセス 1 / area 0 で、PE 向けリンク・サイト LAN（Lo0）・
機器管理 /32（Lo9）を広告済み。デフォルトルートは持たない。

| CE | 顧客/サイト | OSPF router-id | サイト LAN (Lo0) | 機器管理 (Lo9) |
|----|------------|----------------|------------------|----------------|
| RT04 | CUST_A site1 | 4.4.4.4 | 172.16.1.1/24 | 10.99.4.1/32 |
| RT05 | CUST_A site2 | 5.5.5.5 | 172.16.2.1/24 | 10.99.5.1/32 |
| RT06 | CUST_B site1 | 6.6.6.6 | **172.16.1.9/24** | 10.99.6.1/32 |
| RT07 | CUST_B site2 | 7.7.7.7 | **172.16.2.9/24** | 10.99.7.1/32 |

PE との接続セグメントは 01 と同一（RT04: 192.168.1.0/30, RT05: 192.168.2.0/30,
RT06: 192.168.11.0/30, RT07: 192.168.12.0/30。いずれも CE=.1 / PE=.2）。

## 設計仕様（この通りに構成すること）

1. **PE-CE OSPF** — 各 PE で顧客ごとに OSPF プロセスを立て、CE と area 0 で隣接を張る。

   | VRF | PE 側 OSPF プロセス番号 | area | PE 側 router-id |
   |-----|------------------------|------|-----------------|
   | CUST_A | 10 | 0 | Loopback0 と同値（PE1=1.1.1.1 / PE2=3.3.3.3） |
   | CUST_B | 20 | 0 | 同上 |

2. **相互再配布** — 各 VRF で OSPF ⇄ MP-BGP(VPNv4) を相互に再配布し、
   サイト間の到達性を成立させる。
3. **広告制御** — VPNv4 へ広告してよいのは **172.16.0.0/16 内の LAN（/24 まで）のみ**。
   機器管理 10.99.0.0/16 を含む他のプレフィックスは VPNv4 に載せないこと。
   **prefix-list を参照する route-map** で実装すること（名称は任意）。
4. **MSS 調整** — 両 PE の **CE 向けインタフェース（計 4 つ）**で TCP MSS を **1452** に
   調整すること。
5. スタティックルートは使用しないこと（撤去済みのものを戻さない）。

## 到達目標（最終状態）
- **CUST_A**: RT04 から `ping 172.16.2.1 source 172.16.1.1` が成功。
- **CUST_B**: RT06 から `ping 172.16.2.9 source 172.16.1.9` が成功。
- 対向サイトの LAN 経路が **CE に OSPF 経路として**現れる。
- 機器管理 /32 は**収容 PE の VRF には在る**が、**対向 PE・対向サイトには無い**。
- 顧客間の分離は 01 と同様に維持。

## 制約
- CE（RT04〜RT07）の設定変更禁止。
- コアの既存構成（OSPF 1 / LDP / BGP セッション / VRF 定義・IF 収容）を壊さないこと。
  P(RT02) は引き続き BGP を持たないこと。
- 管理用 VRF `MGMT` と Ethernet0/3 には触れないこと。
