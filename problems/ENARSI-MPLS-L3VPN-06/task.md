# 問題 ENARSI-MPLS-L3VPN-06 : MPLS L3VPN 発展 — Extranet: 共有サービス VRF と選択的 Route-Target 設計（難易度5）

## シナリオ

あなたは SP（AS 65000）のエンジニアです。MPLS L3VPN サービス（CUST_A / CUST_B）は
ENARSI-MPLS-L3VPN-04 で構築した完成形で**安定稼働中**です
（PE-CE eBGP・CUST_B は全サイト AS 65200 で as-override 収容・
受信制御 RM-CE-IN により顧客からは 172.16.0.0/16 の LAN のみ受け入れ・
機器管理 10.99.0.0/16 は遮断）。

このたび SP は**共有サービス基盤**（DNS / NTP / 監視、172.30.0.0/24）の提供を開始し、
**両顧客が利用を契約**しました。基盤収容用の SVC-PE（RT08）と SVC-CE（RT09）は
増設済みで、SVC-PE はコア（OSPF/LDP/VPNv4 メッシュ）への参加まで完了しています。

> **新サービス契約チケット — 共有サービスへの接続**
>
> 1. 両顧客の**全サイトから共有サービス（172.30.0.0/24）への相互到達**を提供すること。
>    各顧客は共有サービス利用用のセグメントを各サイトに用意し、**広告済み**である:
>    CUST_A = **10.65.0.0/16 内の /24**（site1: 10.65.1.0/24, site2: 10.65.2.0/24）
>    CUST_B = **10.66.0.0/16 内の /24**（site1: 10.66.1.0/24, site2: 10.66.2.0/24）
> 2. **顧客間（CUST_A ↔ CUST_B）の分離は引き続き完全に維持**すること。
>    片顧客の経路が他顧客の VRF に現れることは重大事故として扱う。
> 3. **共有サービス VRF に顧客のサイト LAN（172.16.0.0/16）を持ち込まないこと。**
>    両顧客は 172.16 帯を**重複使用**しており、共有 VRF に持ち込むと経路が衝突し
>    片顧客が予告なく不達になる。共有 VRF に収容してよいのは**非重複の
>    利用セグメント（10.65 / 10.66 帯）のみ**。
> 4. 顧客の機器管理 10.99.0.0/16 は**引き続き SP 網に受け入れない**こと（既存方針の継続）。
> 5. CE（RT04〜RT07・RT09）の設定変更は不可。既存の顧客間通信
>    （172.16 の各サイト間・as-override 収容）に影響を与えないこと。

```
 CUST_A CE1                                          CUST_A CE2
   RT04 ---\                                        /--- RT05
  (AS 65101) RT01 ========= RT02 ========= RT03  (AS 65102)
   RT06 ---/ (PE1)          (P)          (PE2) \--- RT07
  (AS 65200)                 |                     (AS 65200)
 CUST_B CE1                RT08 (SVC-PE)            CUST_B CE2
                             |
                           RT09 (SVC-CE, AS 65300, 172.30.0.0/24)
```

## 構成（初期状態で投入済み）

- 04 完成形: PE1/PE2 の VRF CUST_A（RD/RT 65000:100）・CUST_B（RD/RT 65000:200）、
  PE-CE eBGP（B は as-override）、RM-CE-IN（172.16.0.0/16 le 24 のみ許可）。
- SVC-PE（RT08, Lo0 8.8.8.8）: コア参加済み（OSPF/LDP/VPNv4 フルメッシュ）。
  **VRF と PE-CE は未構成**。SVC-CE 向け IF（192.168.30.2/30）は暫定で global に IP のみ。
- SVC-CE（RT09, AS 65300）: 172.30.0.0/24 を広告して PE を待ち受け中（Idle）。
- 各顧客 CE: 利用セグメント（Lo2）を**広告済み** — ただし現在は PE の受信制御が
  堰き止めているため SP 網には入っていない。

| CE | 顧客/サイト | AS | サイト LAN (Lo0) | 利用セグメント (Lo2) | 機器管理 (Lo9) |
|----|------------|-----|------------------|----------------------|----------------|
| RT04 | CUST_A site1 | 65101 | 172.16.1.1/24 | **10.65.1.1/24** | 10.99.4.1/32 |
| RT05 | CUST_A site2 | 65102 | 172.16.2.1/24 | **10.65.2.1/24** | 10.99.5.1/32 |
| RT06 | CUST_B site1 | 65200 | **172.16.1.9/24** | **10.66.1.1/24** | 10.99.6.1/32 |
| RT07 | CUST_B site2 | 65200 | **172.16.2.9/24** | **10.66.2.1/24** | 10.99.7.1/32 |

## 設計仕様（この通りに構成すること・変更は PE = RT01/RT03/RT08 のみ）

1. **共有サービス VRF（RT08）** — VRF 名 **SVCS**、RD **65000:300** で新設し、
   SVC-CE 向け IF を収容（PE 側 IP 192.168.30.2/30）、SVC-CE（AS 65300）と eBGP を確立する。
2. **RT 設計** — 使用する RT 値は **65000:300（共有サービス発の経路）** と
   **65000:301（顧客の利用セグメント発の経路）** の 2 つ。配置は要件から設計すること。
3. **利用セグメントの選択的共有** — 顧客 VRF からは **export map を用いて
   利用セグメントにのみ RT 65000:301 を付与**して共有側へ渡すこと
   （SP 標準設計。サイト LAN 172.16 に余計な RT を付けない）。
4. **受信制御の拡張** — RM-CE-IN を、利用セグメントを受け入れるように拡張すること。
   **10.99 帯の遮断は維持**すること。
5. **制約** — CE の設定変更禁止（show / debug は可）。スタティックルート・再配布・
   default-information の使用禁止（SVCS の LAN も BGP で受けること）。

## 到達目標（最終状態）

- 全顧客 CE から共有サービスへ: 例 RT04 `ping 172.30.0.1 source 10.65.1.1` 成功
  （4 CE すべて。**利用セグメントを送信元にすること**）。
- 共有サービスから各顧客の利用セグメントへ到達可能（監視の戻り）。
- SVC-PE の VRF SVCS に **10.65/10.66 の /24 ×4 のみ**が収容され、
  **172.16 帯が存在しない**こと。
- 顧客 VRF: 相互の利用セグメントが**漏れていない**こと（A に 10.66 なし / B に 10.65 なし）。
- 既存サービス無影響: 172.16 のサイト間疎通・as-override・10.99 遮断の維持。

## 考察（採点対象外・解答後に自分の言葉で）

1. チケット要件 3（共有 VRF に 172.16 を持ち込まない）が無かったら何が起きるか。
   重複 prefix を import した共有 VRF の経路はどうなるか。
2. 共有サービス基盤（SVC-CE）は両顧客の利用セグメントに到達できる。では
   CUST_A のホストが共有基盤を「踏み台」に CUST_B へ到達することは可能か。
   経路の観点から説明せよ。
