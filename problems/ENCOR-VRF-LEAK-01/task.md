# 問題 ENCOR-VRF-LEAK-01 : VRF-lite + ルートリーク（難易度6）

## シナリオ
PE ルータ **RT01** に 2 社の顧客（**RED** / **BLUE**）を VRF で収容している。両顧客は
互いに**完全分離**したいが、社内の**共有サービス（DNS/監視など）= `100.64.0.1`** には
**両顧客とも到達**できる必要がある。共有サービスは RT01 上の **SHARED VRF** に置く。

顧客拠点ルータ **RT02** は設定済み（**変更不可**）。あなたは **RT01 のみ**を構成し、
VRF・PE-CE 経路・**選択的なルートリーク**を実現する。

## トポロジ
```
   RED 顧客拠点 Lo1 172.16.1.1/24                 ┌─ SHARED VRF
        (RT02, VRF RED) ── Et0/0 ═══ Et0/0 ──┐    │  共有サービス
                                              RT01 ┤  Lo  100.64.0.1/32
        (RT02, VRF BLUE) ─ Et0/1 ═══ Et0/1 ──┘    └─ (PE / 変更対象)
   BLUE 顧客拠点 Lo2 172.16.2.1/24
```

| ルータ | 役割 | VRF / 用途 |
|--------|------|-----------|
| RT01 | PE（**変更対象**） | RED(Et0/0=10.1.12.1/30) ／ BLUE(Et0/1=10.2.12.1/30) ／ SHARED(共有サーバ Lo 100.64.0.1/32) |
| RT02 | 顧客拠点 CE（**変更不可**） | RED(Lo1 172.16.1.1/24, Et0/0=10.1.12.2) ／ BLUE(Lo2 172.16.2.1/24, Et0/1=10.2.12.2)。各 VRF は RT01 へデフォルト |

## 到達目標 — RT01 のみ
1. VRF **RED / BLUE / SHARED** を定義し、各インタフェース／共有サーバ Loopback を所属させる。
2. PE-CE 経路：RT01 が RED 顧客 `172.16.1.0/24`・BLUE 顧客 `172.16.2.0/24` を学習する。
3. **ルートリーク**：
   - 共有サービス `100.64.0.1` を **RED と BLUE の両方**から到達可能にする。
   - **RED と BLUE は相互に到達不可**（顧客間は完全分離）のままにする。

## 制約
- 変更してよいのは **RT01 のみ**。RT02 は変更不可。
- 「RED↔BLUE は遮断、両者↔SHARED は許可」を**ルートターゲット等の制御**で実現すること
  （全 VRF を素通しにする・分離を壊す解は不可）。

## アクセス・採点
SSH `SUZUKI / CCNP`（mgmt は build 時に割当）。
```
ansible-playbook playbooks/grade.yml -e problem=ENCOR-VRF-LEAK-01 --vault-password-file <(printf 'CCNP\n')
```
※ 採点は `show vrf` / `show ip route vrf <名>` に加え、RT02 から **`ping vrf RED/BLUE 100.64.0.1`（到達）** と
**`ping vrf RED 172.16.2.1`（分離＝失敗）** を能動確認します。
