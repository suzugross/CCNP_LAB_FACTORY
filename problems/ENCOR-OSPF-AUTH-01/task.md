# 問題 ENCOR-OSPF-AUTH-01 : OSPF インタフェース認証 (MD5) でリンクを守る

## シナリオ
キャンパスネットワークの監査で「OSPF が無認証で動いている、悪意のあるルータが
接続されると LSA を注入できる」と指摘された。トランジット 3 ルータの間を結ぶ
2 本のリンクに **MD5 認証** を有効化し、不正なルータからの隣接形成を防ぐ。

## トポロジ
```
[Lo0=1.1.1.1]       [Lo0=2.2.2.2]       [Lo0=3.3.3.3]
RT01 ─────────── RT02 ──────────────── RT03
       10.0.12.0/30      10.0.23.0/30
        area 0             area 0
```

- RT01 ↔ RT02: `10.0.12.0/30` (RT01=.1 / RT02=.2) **area 0**
- RT02 ↔ RT03: `10.0.23.0/30` (RT02=.1 / RT03=.2) **area 0**
- 初期状態: 全ノードが area 0 で OSPF 隣接 FULL、**認証なし**

## 到達目標
- 2 本の inter-router リンクに **MD5 認証**を有効化する。
- 認証適用後も、両方の OSPF 隣接が **FULL** で維持されること（鍵が一致していれば落ちない）。

## 制約
- 認証は **インタフェース単位**で適用すること（area 単位ではない）。
- 認証パラメータ（採点が固定値で判定）:
  - Key ID = **1**
  - Key string = **`ENCOR2026`**
  - 認証タイプ = **message-digest** (MD5)
- OSPF プロセス ID (1) や area 番号 (0) は変更しない。
- 既存のインタフェース IP / `network` 文は変更しない。

## アクセス
- RT01: `10.1.10.11` / RT02: `10.1.10.12` / RT03: `10.1.10.13`（SSH, admin/CCNP）

## 採点
```
ansible-playbook playbooks/grade.yml -e problem=ENCOR-OSPF-AUTH-01 --vault-password-file <(printf 'CCNP\n')
```
