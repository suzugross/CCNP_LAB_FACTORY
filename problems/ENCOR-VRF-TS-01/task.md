# 障害対応 ENCOR-VRF-TS-01 : VRF-lite ルートリーク（共有サービス不通 / 分離）

## 状況
PE ルータ **RT01** に 2 社（**RED** / **BLUE**）を VRF 収容し、共有サービス
`100.64.0.1`（**SHARED VRF**）へ **両社から到達**できる設計（RED↔BLUE は分離）。
構築作業の後、**共有サービスに到達できない**という申告が複数上がっています。

顧客拠点 **RT02 は設定済み・変更不可**。調査・修正は **RT01 のみ**。

```
 RED 拠点 Lo1 172.16.1.1/24
   (RT02,VRF RED) ─Et0/0═Et0/0─┐         ┌ SHARED: 共有サーバ Lo100 100.64.0.1/32
                               RT01(PE) ──┤
   (RT02,VRF BLUE)─Et0/1═Et0/1─┘         └ (変更対象)
 BLUE 拠点 Lo2 172.16.2.1/24
```

## 受付チケット
> - 「**RED 拠点（`172.16.1.1`）から共有サービス `100.64.0.1` に到達できない。**」
> - 「**BLUE 拠点（`172.16.2.1`）からも `100.64.0.1` に到達できない。**」
>
> 切り分けて原因を特定し、恒久的に復旧してください。**原因は 1 か所とは限りません。**
> なお **RED↔BLUE の分離は維持**すること（顧客間が通ってしまう状態も不具合です）。

## 構成台帳
| ルータ | 役割 | VRF / IF |
|---|---|---|
| RT01 | PE（**変更対象**） | RED(Et0/0=10.1.12.1/30) / BLUE(Et0/1=10.2.12.1/30) / SHARED(Lo100 100.64.0.1/32) |
| RT02 | CE（**変更不可**） | RED(Lo1 172.16.1.1/24,10.1.12.2) / BLUE(Lo2 172.16.2.1/24,10.2.12.2)・各VRF は RT01 へデフォルト |

## 完了条件
1. **RED・BLUE の両拠点から `100.64.0.1` へ到達**できる。
2. **RED↔BLUE は相互到達不可**（完全分離）を維持。
3. RT01 の各 VRF RIB がリーク後の正しい状態（SHARED に両顧客 /24、RED/BLUE に `100.64.0.1/32`）。

## アクセス・採点
SSH `SUZUKI / CCNP`（mgmt は build 時に割当）。
```
ansible-playbook playbooks/grade.yml -e problem=ENCOR-VRF-TS-01 --vault-password-file <(printf 'CCNP\n')
```
> 起点：`show ip route vrf RED|BLUE|SHARED` / `show vrf` / `show bgp vpnv4 unicast all`。
> ヒント：RED と BLUE は **同じ「共有サービスに行けない」症状でも原因の層が違います**。
> 「行きの経路が無い」のか「戻りの経路が無い」のか、各 VRF の RIB を双方向で見比べること。
