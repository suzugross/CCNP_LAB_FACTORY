# 問題 ENARSI-MPLS-L3VPN-04 : MPLS L3VPN 応用 — PE-CE eBGP 化・同一 AS 顧客の収容・受信広告制御（難易度4）

## シナリオ
ENARSI-MPLS-L3VPN-01 で構築した L3VPN サービス（CUST_A / CUST_B）は**稼働中**です。
コア（OSPF・LDP・MP-BGP VPNv4・VRF 収容）は構成済み・正常動作しています。

顧客 2 社から**契約変更チケット**が届きました。SP 側エンジニアとして PE を変更してください。

> **チケット 1（両顧客共通）— PE-CE ルーティングの BGP 化**
> 運用標準化のため、PE-CE の経路交換を **eBGP** に切り替えたい。CE 側は
> BGP の設定・稼働を開始済み（下表の AS 番号）。PE 側の設定をお願いしたい。
>
> **チケット 2（CUST_B のみ）— AS 番号の全社統一**
> 当社（CUST_B）は社内標準により**全サイトを AS 65200 に統一**した。
> この AS 設計のままで**サイト間の相互到達**を成立させてほしい。
> なお CE 機器の設定変更は受け入れられない（顧客管理・変更禁止）。
>
> **チケット 3（両顧客共通）— 広告は必要最低限に**
> CE には**機器管理用プレフィックス（10.99.0.0/16 内の /32）**が設定されており、
> BGP でも広告してしまっている。これは**各サイト内に閉じるべき**もので、
> **SP 網に受け入れてはならない**（収容 PE の VRF にも載せないこと）。
> VPN で運ぶのは**サイト LAN（172.16.0.0/16 内の /24 まで）のみ**とすること。

```
 CUST_A CE1                                          CUST_A CE2
   RT04 ---\                                        /--- RT05
  (AS 65101) RT01 ========= RT02 ========= RT03  (AS 65102)
   RT06 ---/ (PE1)          (P)          (PE2) \--- RT07
  (AS 65200)                                     (AS 65200)
 CUST_B CE1  └──────── SPコア (構成済み) ───────┘  CUST_B CE2
```

## 構成（初期状態で投入済み）

### コア（構成済み・正常動作）
- OSPF 1 / LDP(コアのみ)、PE1–PE2 の iBGP VPNv4（AS 65000, Lo0 間）、
  VRF CUST_A（RD 65000:100 / RT 65000:100）・CUST_B（RD 65000:200 / RT 65000:200）
  定義済み、CE 向け IF は VRF 収容・IP 設定済み。アドレスは 01/02 と同一。
- **PE-CE のルーティングは未設定** = 現在サイト間は不通。
  BGP の `address-family ipv4 vrf` も未設定（あなたが構成する）。

### CE（RT04〜RT07 = 顧客管理機器・設定済み・変更禁止）
各 CE は BGP で PE（AS 65000）とのセッションを待ち受け中。サイト LAN（Lo0）と
機器管理 /32（Lo9）を network 文で広告済み。

| CE | 顧客/サイト | AS | router-id | サイト LAN (Lo0) | 機器管理 (Lo9) |
|----|------------|-----|-----------|------------------|----------------|
| RT04 | CUST_A site1 | **65101** | 4.4.4.4 | 172.16.1.1/24 | 10.99.4.1/32 |
| RT05 | CUST_A site2 | **65102** | 5.5.5.5 | 172.16.2.1/24 | 10.99.5.1/32 |
| RT06 | CUST_B site1 | **65200** | 6.6.6.6 | **172.16.1.9/24** | 10.99.6.1/32 |
| RT07 | CUST_B site2 | **65200** | 7.7.7.7 | **172.16.2.9/24** | 10.99.7.1/32 |

PE との接続セグメントは 01/02 と同一（RT04: 192.168.1.0/30, RT05: 192.168.2.0/30,
RT06: 192.168.11.0/30, RT07: 192.168.12.0/30。いずれも CE=.1 / PE=.2）。

## 設計仕様（この通りに構成すること）

1. **PE-CE eBGP** — 各 PE の BGP（AS 65000）に **VRF 単位の address-family** を構成し、
   CE と eBGP セッションを確立する。
2. **経路交換は BGP のみで完結させる** — スタティックルート・他プロトコルへの/からの
   **再配布は一切使用しない**こと。
3. **広告制御（受信段階で破棄）** — 機器管理 10.99.0.0/16 系の経路は **PE が CE から
   受信する段階で破棄**し、収容 PE の VRF にも載せないこと。受け入れてよいのは
   **172.16.0.0/16 内の /24 まで**。**prefix-list を参照する route-map** を
   **neighbor 単位の in 方向**に適用して実装すること（名称は任意）。
4. **CUST_B の同一 AS 収容** — チケット 2 の AS 設計（両サイト 65200）のまま
   サイト間の相互到達を成立させること。**CE の設定変更は禁止**。

## 到達目標（最終状態）
- **CUST_A**: RT04 から `ping 172.16.2.1 source 172.16.1.1` が成功。
- **CUST_B**: RT06 から `ping 172.16.2.9 source 172.16.1.9` が成功。
- 対向サイトの LAN 経路が **CE に BGP 経路として**現れる。
- 機器管理 /32 は **CE 自身とそのサイト内に閉じる**（収容 PE の VRF・対向 PE・
  対向サイトのいずれにも無い）。
- 顧客間の分離は 01/02 と同様に維持。

## 制約
- CE（RT04〜RT07）の設定変更禁止。
- スタティックルート・再配布（redistribute）の使用禁止（仕様 2）。
- コアの既存構成（OSPF 1 / LDP / VPNv4 セッション / VRF 定義・IF 収容）を壊さないこと。
  P(RT02) は引き続き BGP を持たないこと。
- 管理用 VRF `MGMT` と Ethernet0/3 には触れないこと。
