# 設計書: ENARSI-BGP-POLICY-01 — BGP属性複合・経路ポリシー構築問

status: **設計のみ（未実装）** / 作成 2026-07-04
方向性(1)「BGP属性ドリル群の複合問化」の第1問。単発ドリル10問
（community / LP / MED / NHSELF / aggregate / prefix-list / route-map 等）の
知識を1問に統合し、経路選択プロセスの理解を問う。

## 1. 出題コンセプト

- 種別: **構築＋ポリシー設計**（TSではない）。difficulty **5** / 想定 60–90分
- 顧客AS（3台）だけを設定させ、ISP側（3台・変更不可）が「動かせない現実」
  として制約を与える。実務の「上流はいじれない」状況を再現する。
- 核となるひねり:
  - **weight による手抜きを構造的に殺す**——採点が「内部ルータ RT03 の経路
    選択」を見るため、伝搬しない weight では要件を満たせず、
    community→LP の AS全体ポリシーが必然になる（問題文では一切示唆しない。
    [[ccnp-problem-hint-policy]] 準拠）。
  - **in/out の対称設計**——outbound は LP、inbound は MED と、方向で別の
    属性を使い分けさせる。
  - 経路選択9段のうち weight/LP/origin無関係/MED の使いどころを体感させる。

## 2. トポロジ（6ノード・IOL・全ルータ次数≤3）

```
                    AS 65099 (上流 Internet)
                         RT06  ←beacon経路の発生源（変更不可）
                        /    \
              AS 65020 /      \ AS 65030
               (ISP-A)         (ISP-B)
                 RT04           RT05      ←ともに変更不可
                /    \             \
            RT01      RT02 ---------+
（edge-A・主系）\      /（edge-B・ISP-A待機系＋ISP-B接続）
                 RT03
            （内部コア・eBGPなし）
                    AS 65010（顧客・設定対象）
```

| リンク | 用途 | セグメント |
|--------|------|-----------|
| RT01–RT04 | eBGP 主系 | 100.64.14.0/30 |
| RT02–RT04 | eBGP ISP-A 待機系 | 100.64.24.0/30 |
| RT02–RT05 | eBGP ISP-B | 100.64.25.0/30 |
| RT04–RT06 | ISP-A 上流 | 100.64.46.0/30 |
| RT05–RT06 | ISP-B 上流 | 100.64.56.0/30 |
| RT01–RT03 | AS内 | 10.10.13.0/30 |
| RT02–RT03 | AS内 | 10.10.23.0/30 |

RT01–RT02 直結リンクは**置かない**（内部トラフィックが RT03 を必ず経由
= RT03 の経路選択が採点の急所になる）。全ルータ Eth0/3=MGMT、次数≤3 で
現行 device_profiles のまま収容可。

### アドレス・プレフィックス設計

- Lo0: RT0x = x.x.x.x/32（リポジトリ慣例どおり）
- 顧客保有: **198.51.100.0/24**。実体は RT03 の Lo10=198.51.100.1/25,
  Lo11=198.51.100.129/25（→集約課題の素材）
- beacon（RT06 の Lo で発生・両ISPへ広告済み）:
  - BEACON-A **192.0.2.0/24**（通常トラフィックの代表）
  - BEACON-B **203.0.113.0/24**（例外ポリシー対象）
  - BEACON-C **198.18.100.0/24**（通常その2・採点の冗長性用）
- **汚染経路（RT04 が意図的にリーク・変更不可）**:
  - 172.16.66.0/24（RFC1918）
  - 198.18.200.16/28（/24 より長い）
  → 受信フィルタ課題の素材。static→BGP redistribute で RT04 に焼き込む。
- 10.1.10.0/26（MGMT）とは全て非重複。

## 3. 初期状態（day0 に焼き込むもの）

| ノード | 初期投入済み | 未設定（=受験者の仕事） |
|--------|-------------|------------------------|
| RT01/RT02 | IF/IP・OSPF(area0, Lo0含む)・**eBGP セッションのみ**（neighbor 定義済み・Established・ポリシー皆無） | iBGP 全て・全ポリシー |
| RT03 | IF/IP・OSPF・Lo10/11（/25×2） | iBGP・(必要なら)network文 |
| RT04 | eBGP×3(RT01/RT02/RT06)・beacon中継・汚染経路リーク | —（変更不可） |
| RT05 | eBGP×2(RT02/RT06)・beacon中継 | —（変更不可） |
| RT06 | beacon Lo×3・eBGP×2 | —（変更不可） |

- eBGP を事前確立しておくのは「セッション張り」でなく「ポリシー設計」に
  時間を使わせるため。iBGP は白紙（Lo0 ピア＋next-hop-self を書かせる）。
- ISP側に `bgp deterministic-med` は入れない（IOS 既定の同一隣接AS間MED比較
  で課題6は成立する）。

## 4. 受験者要件（task.md 骨子・ヒント控えめ版）

> あなたは AS65010 の運用者である。ISP-A(AS65020)・ISP-B(AS65030)・上流
> (AS65099) の機器には一切ログインできない。AS65010 の3台のみ設定せよ。
>
> 1. AS65010 内の BGP を完成させること。内部ピアリングは **Loopback0 間**で
>    行い、物理リンク障害1本でセッションが落ちない構成とすること。
>    RT03（eBGPを持たない）からもインターネット経路が利用できること。
> 2. 外部から学習した経路には、学習元が判別できるよう community を付与する
>    こと: ISP-A 由来=**65010:100** / ISP-B 由来=**65010:200**。この
>    community は AS65010 内の全ルータで確認できること。
> 3. インターネット向けトラフィックは、**全ルータにおいて** RT01–ISP-A
>    リンクから出ること（RT02 の ISP-A リンクは待機系）。
> 4. ただし **203.0.113.0/24** 宛だけは、**全ルータにおいて** RT02–ISP-B
>    リンクから出ること。
> 5. 顧客プレフィックスは **198.51.100.0/24 の集約のみ**を外部へ広告する
>    こと（/25 が外部に見えてはならない）。集約は両エッジから行うこと。
> 6. インターネット側から 198.51.100.0/24 宛のトラフィックが **ISP-A 網内
>    では RT01 側リンクに入る**よう、AS65010 から制御すること
>    （AS-path 長を変えてはならない）。
> 7. ISP から受信する経路のうち、**RFC1918 と /24 より長いプレフィックス**
>    は BGP テーブルに載せないこと。
> 8. 完了時、RT03 から全 beacon への到達性があること。
>
> 制約: RT04/RT05/RT06 は変更不可。AS65010 側でも `ip route`（静的経路）の
> 追加は禁止。

- 要件6の「AS-path 長を変えてはならない」で prepend を封じ、MED を強制
  （封じないと prepend が正当な別解になり MED 課題が消えるため。ここだけは
  意図駆動の制約文で縛る）。
- 要件3+4の「全ルータにおいて」が weight 封じの仕掛け（本文では説明しない）。
- 要件2は send-community 忘れの検出を兼ねる（RT03 で tag が見えない）。

## 5. 採点設計（100点・状態ベース）

方針: config 文字列マッチは使わず、**BGPテーブル/RIB/到達性**で採点
（[[ccnp-global-invariant-grader]] の思想。route-map の書き方は自由＝
prefix-list 直マッチで LP を付けても、結果と community 要件を満たせば正解。
実装スタイルは採点後レビューで講評する）。

| # | 配点 | 判定 | 取得元（Genie/SSH） |
|---|------|------|--------------------|
| 1 | 15 | RT01/02/03 の iBGP が **Lo0同士**で Established（`show bgp summary` の neighbor が x.x.x.x）＋ next-hop-self 相当（RT03 の外部経路 next-hop が 1.1.1.1/2.2.2.2） | show bgp ipv4 uni summary / show ip route |
| 2 | 15 | RT03 の BGPテーブルで beacon 経路に 65010:100、ISP-B由来ベスト経路に 65010:200 が付着 | show bgp ipv4 uni <pfx> |
| 3 | 15 | RT01/02/03 全てで 192.0.2.0/24 と 198.18.100.0/24 のベストパスが RT01–RT04 リンク経由（RT03では next-hop 1.1.1.1） | show bgp / show ip route |
| 4 | 15 | RT01/02/03 全てで 203.0.113.0/24 のベストパスが RT02–RT05 経由（RT03では next-hop 2.2.2.2、RT01では iBGP経由で RT02） | 同上 |
| 5 | 10 | RT06 の BGPテーブルに 198.51.100.0/24 が存在し **/25 が不在**。RT04 から見て両エッジ発の /24 が2経路ある（両エッジ集約） | RT06/RT04 を読取採点（変更不可≠読取不可） |
| 6 | 10 | RT04 の 198.51.100.0/24 ベストパスが RT01 セッション由来で、**MED差**が決め手（両経路の AS-path 長が等しいことも確認=prepend検出） | show bgp ipv4 uni 198.51.100.0 on RT04 |
| 7 | 10 | RT01/RT02 の BGPテーブルに 172.16.66.0/24・198.18.200.16/28 が**不在**（かつ beacon は健在=過剰フィルタ検出） | show bgp |
| 8 | 10 | RT03 から 3 beacon へ ping 成功＋ 192.0.2.1 への traceroute に 100.64.14.x が、203.0.113.1 へのそれに 100.64.25.x が現れる | ping/traceroute (exec) |

- 部分点: 各項目内で「一部ルータのみ満たす」場合は按分（既存 grade.py の
  条件配列で表現可能）。
- 採点前に `clear ip bgp * soft` 相当の反映漏れで落ちるのを防ぐため、
  **チェック7/3/4 は リトライ付き（最大2回・間隔30s）** とする。

### 想定別解と扱い

| 別解 | 結果 | 扱い |
|------|------|------|
| weight で RT01/RT02 個別に制御 | RT03 の bestpath が要件3/4を満たせない | 構造的に不正解（狙いどおり） |
| LP を community でなく prefix-list 直で設定 | 要件2(tag付着)と結果を満たせば通る | **正解**。レビューで「community基準の利点」を講評 |
| 例外経路を RT03 の inbound route-map で捻じ曲げ | RT01 側の bestpath が要件4を満たさない可能性大 | ほぼ構造的に阻止 |
| 集約を RT03 で実施 | RT04 に2経路できず #5後半で減点 | 部分点で表現 |
| distribute-list で汚染経路除去 | 状態は同じ | 正解（手段は問わない） |

## 6. 実機検証チェックリスト（実装フェーズで実施）

1. day0 のみで ISP 側3台の経路交換・beacon/汚染経路の広告が完成すること
2. 模範解答を流して 100/100（採点8項目全通過）
3. **無設定（初期状態のまま）で 15点未満**であること（初期状態リーク検査）
4. weight だけで解いた「罠解答」が #3/#4 で確実に減点されること
5. MED 課題: IOL の同一隣接AS間 MED 比較が既定で効くことの確認
   （効かない場合は RT04 に `bgp always-compare-med` を焼き込む設計変更）
6. 集約課題: aggregate-address 後の /25 suppress が RT04/RT06 まで届くこと
7. 採点リトライ動作（BGP 収束 30–60s を許容できるか）

## 7. 既知リスク・未決事項

- **IOL の MED 既定動作**は実機確認が必須（チェックリスト5）。ここが今設計
  唯一の「実機で覆り得る」ポイント。覆った場合も RT04 焼き込みで吸収可能。
- 要件6の採点で「MED差が決め手」を Genie 出力から機械判定する部分は、
  `show bgp` の bestpath 理由文字列のパース次第で簡略化（MED値の大小比較のみ）
  に落とす可能性あり。
- mgmt 台帳: 6台なので .11–.16。**将来生成器化する際は 10.1.10.{11+i}
  直書きにしない**こと（/26化後の既知の落とし穴）。
- 生成器化(seed でプレフィックス/community値/例外beacon をランダム化)は
  本問の実機100点確認後に着手（gen_bgp_complex_ts.py の骨格を流用可能）。
