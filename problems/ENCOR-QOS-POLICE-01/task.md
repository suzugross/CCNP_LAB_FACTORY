# ENCOR-QOS-POLICE-01: 暴走するバックアップ転送を 1Mbps に抑えろ（効果実測つき）

難易度: ★★★☆☆（3/6） / 配点: 100点 / 対象: ENCOR (350-401) QoS

QoS 体感シリーズ第2問。学習順序: **CLASS-01 → POLICE-01 → LLQ-01**。

## シナリオ

拠点1 のバックアップシステムが営業時間中にも WAN へ全力で転送を始めてしまい、
他の業務通信を圧迫している。バックアップベンダー側の帯域設定は当てにできない。

そこで**ネットワーク側で強制的に**、バックアップ転送だけを 1Mbps に
制限することになった。超過分はバッファせずその場で破棄してよい
（バックアップは TCP なので、送信側が勝手に速度を落とす——それを体感するのも本問の狙い）。

```
PC01 ── RT01 ──[ WAN ]── RT02 ── PC02
拠点1          10.1.12.0/30       拠点2
 .10   E0/0        E0/1    E0/1   E0/0   .20
192.168.10.0/24              192.168.20.0/24
```

- PC01 / PC02: Ubuntu（ログイン `SUZUKI` / `CCNP`）。iperf3 導入済み。
  PC02 では iperf3 サーバが常駐起動している。
- RT01 / RT02 の IP・経路は設定済み（**変更禁止**）。

## 対象トラフィック

| 社内システム | トラフィックの実体 | 扱い |
|--------------|--------------------|------|
| バックアップ転送 | **TCP 宛先ポート 5201** | **1Mbps に制限（超過は破棄）** |
| それ以外（音声 UDP・死活監視 ICMP 等） | 上記以外すべて | **影響を与えないこと** |

## 要件

1. RT01 の **WAN 出力**で、バックアップ転送（TCP 宛先 5201）だけを
   **1Mbps に制限**せよ。超過分は破棄すること（バッファリングしない）。
2. バックアップ以外のトラフィックには**一切影響を与えない**こと。
3. 設定変更は **RT01 のみ**（RT02・PC の変更は禁止。既存のアドレス・経路も変更禁止）。
4. class-map / policy-map / ACL 等の**名前は任意**（採点は名前でなく状態と効果を見る）。

## 体感手順 — 「制限される」「送信側が順応する」「巻き添えがない」を数値で見る

PC01 から:

```
# 測定A: バックアップ転送 (TCP 5201) — 実装前後のスループット比較
iperf3 -c 192.168.20.20 -t 10

# 測定B: バックアップ転送中の死活監視 — 巻き添え確認
#   (ターミナル2本 or & で、iperf3 実行中に)
ping -c 20 -i 0.2 192.168.20.20

# 測定C: 音声ストリーム (UDP 5Mbps) — 制限対象外の確認
iperf3 -c 192.168.20.20 -u -b 5M -l 1200 -t 10
```

RT01 側の観測: `show policy-map interface Ethernet0/1`
— conformed（通過）/ exceeded（破棄）のパケット数とレートが読める。

**記録表**（実装前後で埋める）:

| 測定 | 実装前 | 実装後 |
|------|--------|--------|
| A: TCP スループット | Mbps | Mbps |
| B: 転送中の ping (RTT avg / loss%) | ms / % | ms / % |
| C: UDP 5Mbps の受信レート / loss% | Mbps / % | Mbps / % |
| RT01: conformed / exceeded packets | - | / |

考察ポイント: TCP は破棄されているのに、なぜ実装後のスループットは
「ほぼ 1Mbps」に落ち着くのか（誰が速度を合わせているのか）。
UDP なら同じ制限でどうなるか（→ LLQ-01 で体感する）。

## 採点

- 採点は自動（構成の状態 + 効果の実測）。採点器も PC01 からトラフィックを流す。
- 目安時間: 30〜45 分。
