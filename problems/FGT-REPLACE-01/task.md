# FGT-REPLACE-01: ASA→FortiGate リプレース — 現行機の config を読み解き、移行せよ

あなたはネットワーク更改案件の担当者です。ある拠点で稼働中の **Cisco ASA
（Active/Standby の HA ペア）が EOL** を迎え、**FortiGate 1台**へのリプレースが
決まりました。あわせて拠点の LAN/DMZ は**新しいアドレス設計**へ移行します。

手元にあるのは、現行 ASA の running-config（**[running-config.txt](running-config.txt)**・
紙面のみ。現行機は触れません）と、下記の新環境情報だけです。
**現行 config から「この拠点で許可されている通信と NAT」を読み解き、
同じサービスレベルを FortiGate 上に再現してください。**

★本ラボは**体験型**です。📋観察チェックポイントと🤔考察課題つき。
FGT の構築は GUI（https://10.1.10.11）/CLI どちらでも構いません。
ASA の各要素を FortiGate の**何で**実現するかを調べて決めるのが本問の核心です
（FGT-FW-BASIC-01 / FGT-IPSEC-01 を先にクリアしていることを推奨）。

## 新環境トポロジ（あなたが触るのは FGT のみ・他は据付）

```
 あなたのPC ──┐(管理GUI/同一LAN)
              ├─[SW]── port3 ┌─────┐ port1 ──(WAN)── ISP1 ── INET ─┬─ SRV
 USER-PC ─────┘   10.1.10.0/24│ FGT │                             │ 198.51.100.100
                              └─────┘ port2 ──(DMZ)── DMZ-SRV     └─(他拠点)
                                                       172.16.10.10
```

## アドレス読替表（現行 → 新環境）

| 役割 | 現行（ASA config 内） | 新環境（FGT で使う値） |
|---|---|---|
| outside | 172.16.243.0/24（自機 .251） | **203.0.113.0/30**・port1=**.2**・対向 ISP1=.1 |
| inside（社内 LAN） | 172.16.100.0/24 | **10.1.10.0/24**・port3=.11（**設定済み・変更禁止**） |
| dmz | 172.16.254.0/24 | **172.16.10.0/24**・port2=**.1** |
| 公開サーバの実 IP | 172.16.250.1（LB 配下） | **172.16.10.10**（DMZ-SRV・据付） |
| 公開用グローバル IP | 2.2.2.1 | **2.2.2.1（継続使用）** |
| 発信用グローバル IP | 2.2.2.10 | **2.2.2.10（継続使用）** |

- グローバル IP 2ヶ（2.2.2.1 / 2.2.2.10）は**プロバイダから継続割当**。
  新回線（ISP1）側への経路変更は**プロバイダ側で実施済み**
- 現行の LB・L3SW は新環境では**廃止**（サーバは FGT の port2 直下へ集約）

## Phase 1: 現行構成の読み解き（手を動かす前に紙で勝つ）

running-config.txt を読み、次の**要件表**を自分のメモに書き出せ:

1. この ASA が**許可している通信**は何と何か（向き・送信元・宛先・サービスを列挙。
   ★ACL に書かれているものが全てではない — security-level の規則を思い出すこと）
2. **NAT は何が何に変換される**のか（2 つの object network の nat 行それぞれ）
3. route 4本はそれぞれ何のためか。新環境で**残す必要があるもの**はどれか
4. failover ブロックと standby IP は新環境で**どう扱うべき**か

📋 **観察1**: ACL `OUTSIDE-IN` の宛先が、公開グローバル IP（2.2.2.1）ではなく
**実 IP（172.16.250.1）**で書かれている。これは ASA 8.3 以降の仕様である。
自分の言葉で「ASA の ACL と NAT はどちらが先に評価されるか」をメモせよ。

🤔 **考察1**: inside の PC が dmz のサーバへ HTTP アクセスできるか、
config を根拠に Yes/No で答えよ（ヒント: その通信を許可する ACL は存在しない）。

## Phase 2: 移行設計（概念の対応表を自分で作る）

ASA の各要素を FGT のどの機能で実現するか、**対応表を自分で作成**せよ。
最低限、次の行を埋めること（右列を調べて決めるのが課題）:

| ASA | FortiGate では？ |
|---|---|
| security-level による暗黙許可 | ? |
| access-list + access-group | ? |
| object network + nat static | ? |
| object network + nat dynamic（PAT アドレス指定） | ? |
| inspect icmp | ? |
| failover（HA ペア） | ? |
| ワンアームの dot1q サブインターフェース ×3 | ?（★新環境は物理 3 ポート） |

🤔 **考察2-1**: 移行対象外と判断すべき項目はどれか。その理由と、
「本来 FGT で実現するなら何を使うか」もメモせよ。
🤔 **考察2-2**: ASA では route が 4 本あったが、新環境の FGT で
スタティックルートは何本必要か。減る理由は何か。

## Phase 3: FGT 実装

Phase 2 の対応表に従って実装せよ。**採点の都合上、命名と ID は以下に固定**:

| 種別 | 指定 |
|---|---|
| port1 | alias **WAN**・role wan・**管理アクセスは一切開けない** |
| port2 | alias **DMZ**・role dmz・ping のみ許可 |
| アドレスオブジェクト | **LAN-NET**（社内 LAN）/ **DMZ-SRV**（公開サーバ /32） |
| 発信 NAT 用の部品 | 名前 **SNAT-POOL**（2.2.2.10） |
| 公開 NAT 用の部品 | 名前 **SRV-VIP**（2.2.2.1 → 公開サーバ） |
| ポリシー | **ID 1**=社内→外部 / **ID 2**=外部→公開サーバ / **ID 3**=社内→DMZ |

要件（現行 ASA と同じサービスレベル）:

1. 社内 LAN → インターネット: 全サービス可。**送信元は必ず 2.2.2.10** に変換
   （port1 の IP ではない — 現行の PAT 仕様を維持）
2. 外部 → 公開サーバ: グローバル IP **2.2.2.1** 宛の **HTTP と ICMP のみ**可
   （現行 ACL と同じ2行分。それ以外は不可）
3. 社内 LAN → DMZ: 可（現行 ASA で通っていた通信を殺さないこと）
4. DMZ → 社内 LAN: **不可のまま**（現行と同じ）
5. 上記以外の通信を新たに開けないこと

📋 **観察3-1**: ID 1 と 2 を作った時点で、USER-PC から `wget -q -O -
http://172.16.10.10/` を試せ → **失敗する**はず。現行 ASA でこの通信が
通っていたのはなぜか、FGT で通らないのはなぜかを自分の言葉でメモせよ
（これが**リプレース案件で最も多い移行漏れ**の型である）。

🤔 **考察3**: 公開サーバへの ICMP を通すために、公開 NAT の部品を
「TCP 80 のポート転送」で作ってはいけない。なぜか。

## Phase 4: 挙動検証（移行判定試験）

すべての試験は console から実施（USER-PC / SRV / DMZ-SRV とも root）:

📋 **観察4-1**: USER-PC → `ping 198.51.100.100` / `wget -q -O - http://198.51.100.100/`
📋 **観察4-2**: FGT で `diagnose sys session list | grep 2.2.2` — 発信セッションの
`act=snat` 行で **2.2.2.10 への変換**を確認（2.2.2.1 でも port1 IP でもないこと）
📋 **観察4-3**: SRV（外部側）→ `wget -q -O - -T 5 http://2.2.2.1/` と
`ping -c 3 2.2.2.1` の両方が成功。session list の `act=dnat` 行も確認
📋 **観察4-4**: USER-PC → `wget -q -O - http://172.16.10.10/` が成功
📋 **観察4-5**: DMZ-SRV → `ping -c 3 10.1.10.12` が**全滅**（現行と同じ遮断）

## Phase 5: 移行レビュー（実務の締め）

🤔 **考察5-1**: `inspect icmp` に相当する設定を FGT に入れたか？
入れていないのに ICMP の戻りが通るのはなぜか。
🤔 **考察5-2**: 今回のリプレースで「現行より安全になった点」と
「冗長性の観点で後退した点」を1つずつ挙げよ。
🤔 **考察5-3**: 移行判定試験（Phase 4）に「現行 config には無いが実施すべき」
試験を1つ追加するなら何か（ヒント: 要件5）。

## 完成条件（この状態で採点します）

1. 社内→外部が全サービス通り、送信元が **2.2.2.10** になっている
2. 外部→ **2.2.2.1** の HTTP・ICMP が公開サーバに届く
3. 社内→DMZ の HTTP が通る
4. DMZ→社内が遮断されている（DMZ-SRV 自体は生きていること）
5. IF / オブジェクト / NAT 部品 / ポリシー / ルートが Phase 3 の指定どおり

## ログイン・注意

- FGT: console / GUI（https://10.1.10.11）とも admin / CCNPccnp
- ★FGT で `execute factoryreset` 等の初期化コマンドは**絶対に打たないこと**
  （評価ライセンスが消えます）
- USER-PC / SRV / DMZ-SRV: console で root（プロンプトが出ない時は Enter）
- 据付機器（ISP1 / SRV / DMZ-SRV ほか）は変更禁止。DMZ 配線上に他ラボ用の
  据付ルータが同居しているが本問とは無関係・触らないこと
- 設定は自動保存されます（write memory 不要）
