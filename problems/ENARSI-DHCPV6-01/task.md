# 問題 ENARSI-DHCPV6-01 : DHCPv6 の使い分け — stateless / stateful / リレー（難易度5）

## シナリオ
**IPv6 のみ**で運用する社内ネットワーク。データセンタの **RT01 が全社の中央 DHCPv6 サーバ**、
拠点の **RT02 が2つの LAN セグメントを収容するゲートウェイ**である。
セキュリティポリシーの異なる 2 つの LAN に対し、**アドレス自動化の方式を使い分けて**
クライアント（RT03 / RT04 がホスト役）を開通させる。

## トポロジ
```
                RT01 (中央 DHCPv6 サーバ)
                (Lo0 = 2001:DB8:1::1/128)
                  │ Et0/0 = 2001:DB8:12::1/64
                  │
                  │ Et0/0 = 2001:DB8:12::2/64
        ┌──────── RT02 (拠点GW) ────────┐
        │ Et0/1 = 2001:DB8:A::1/64     │ Et0/2 = 2001:DB8:B::1/64
        │                              │
   LAN-A (開発部門)               LAN-B (経理部門)
        │ Et0/0                        │ Et0/0
       RT03 (クライアント)            RT04 (クライアント)
```

- RT01 / RT02 の **IF アドレスと静的経路は投入済み**（相互到達あり・**変更禁止**）。
- RT03 / RT04 の Et0/0 は **未構成**（リンクは up）。

## 到達目標

### 1. LAN-A（開発部門・stateless 運用）
- クライアント (RT03) は **アドレスをルータ広告ベースで自己生成**する
  （`2001:DB8:A::/64` の GUA を自動取得）。
- ただし **DNS サーバ `2001:DB8:1::53` とドメイン名 `ccnp.local` は、
  RT01 の DHCPv6 から自動配布**されること（RT03 に手動設定しない）。
- RT03 はデフォルト経路も**自動取得**し、RT01 Lo0 (`2001:DB8:1::1`) へ到達できること。

### 2. LAN-B（経理部門・stateful 運用）
- クライアント (RT04) のアドレスは **RT01 の DHCPv6 が `2001:DB8:B::/64` から
  払い出してリース管理**する（RT01 の binding に載ること）。
- **クライアントが自己生成した GUA（EUI-64 等）を持ってはならない**
  （棚卸し要件: LAN-B のクライアントのグローバルアドレスは払い出しの1個のみ）。
- DNS `2001:DB8:1::53` / ドメイン `ccnp.local` も DHCPv6 から配布。
- RT04 もデフォルト経路を**自動取得**し、RT01 Lo0 へ到達できること。

### 3. サーバ設計の制約
- **DHCPv6 サーバ機能は RT01 のみ**で動かす（RT02 でサーバ機能を有効化しない）。
- 両 LAN の要求は **RT01 の同一インタフェース (Et0/0) で受けて、LAN ごとに正しい
  内容を返す**こと（LAN-A にアドレスを払い出してはならない）。

## 制約
- RT01 / RT02 の既設 IF アドレス・静的経路の変更・追加は禁止。
- RT03 / RT04 への **手動 GUA・手動デフォルト経路・静的経路の追加は禁止**
  （自動化のための IF 設定のみ可）。
- 4 台とも設定変更可（上記制約の範囲で）。

## ヒント（最小限）
- ホストの挙動（何を自己生成し、何を DHCPv6 に聞くか）は、**GW が出すルータ広告(RA)の
  フラグ**で決まる。仕様どおりの RA を出すのは GW の責務。
- サーバがオフリンクにある LAN で DHCPv6 を使うには何が要るか、v4 と対比して考えること。
